# 笨猫丫丫 OPC 学习陪跑网站设计文档

> 修订说明：已按《产品执行手册 V2.4》及 `yayasmart-demo` 于 2026-06-15 补充修订。第 15-16 节为当前执行基线，与前文冲突时以修订节为准。

## 1. 项目概述

笨猫丫丫 OPC 学习陪跑网站是一个以 NFC 卡片为入口、以 Dify 后端知识库和导师服务为支撑的大学四年学业陪跑平台。学生通过“碰一碰”打开专属网页，从录取后首次激活开始，逐步完成信息录入、资料上传、AI 初步规划、导师匹配、SOP 确认、日常打卡和成长复盘。

产品核心定位：

> 一张 NFC 卡片，开启四年学业陪跑。

网站不只是一个表单或资料查询工具，而是连接学生从“收到录取通知书”到“大学四年成长”的数字入口。它通过“线下 NFC 触发 + 线上动态规划 + Dify 知识库问答 + 导师人工校准”，提供有仪式感、可迭代、可跟踪的长期陪伴体验。

## 2. 建设目标

- 为每张 NFC 卡片建立唯一数字身份，支持不同状态下的自动路由。
- 让新生完成学校、专业、目标、资料等基础信息提交。
- 接入 Dify 后端知识库，对学校政策、培养方案、学生手册、评奖规则等资料进行问答和规划辅助。
- 生成学生个人 V0.1 SOP 草案，并支持导师确认为 V1.0。
- 提供学生专属仪表盘，展示四年路线图、本周任务、红线预警、奖金收益、荣誉榜和打卡记录。
- 支持学生在大学不同阶段修订目标，保留 SOP 历史版本。
- 为导师提供基础工作台，支持查看学生资料、调用知识库辅助分析、确认规划和更新任务。

## 3. 用户角色

### 3.1 学生

主要使用移动端网页访问。通过 NFC 卡片进入网站，完成激活、提交资料、查看规划、日常打卡、提问和目标修订。

### 3.2 导师

主要使用桌面端或平板端后台。查看系统分配的学生，阅读学生上传资料和 AI 摘要，结合自身经验确认 SOP、维护任务、处理目标修订。

### 3.3 管理员

维护 NFC 卡片、学校知识库、导师资料、学生状态、奖金规则、内容审核和系统配置。

### 3.4 Dify 知识库服务

作为 AI 能力和知识库检索后端，负责基于已上传或已维护的学校资料、平台规则、导师方法论等内容，为前端和后台提供问答、摘要、规划建议和任务拆解能力。

## 4. 整体系统框架

系统采用“前端入口 + 后端业务服务 + 数据库 + Dify 知识库”的分层架构。前端负责学生、导师和管理员的交互体验；后端负责鉴权、业务流程、权限控制、任务编排和第三方服务调用；数据库保存结构化业务数据；知识库保存可检索的学校资料、学生资料、平台方法论和导师经验。

### 4.0 技术选型

考虑到服务器初期配置可能为 2 核 4G，系统优先采用轻量、稳定、易部署的 TypeScript 技术栈：

| 层级 | 技术选型 | 说明 |
| --- | --- | --- |
| 前端 | Vue 3 + Vite + TypeScript | 学生端 H5、导师工作台和管理后台均采用前后端分离模式，前端构建后由 Nginx 托管静态文件 |
| 后端 | NestJS + TypeScript | 作为独立后端 API 服务，负责业务流程、权限控制、Dify 代理、文件处理、通知和 SOP 状态管理 |
| 数据库 | PostgreSQL + Prisma | PostgreSQL 保存结构化业务数据，Prisma 负责类型安全的数据访问和迁移管理 |
| 部署 | Nginx + NestJS API | Nginx 托管前端静态资源并反向代理后端接口，NestJS 提供统一 API 服务 |

该选型不使用 Next.js 或 Nuxt 的 SSR 作为主架构，避免在低配置服务器上引入不必要的运行时开销。学生端、导师端和管理端以业务应用为主，对 SEO 依赖较低，因此静态前端 + 独立 API 服务更适合第一阶段落地。

### 4.1 架构总览

```text
学生移动端 H5 / 导师工作台 / 管理后台
   ↓
前端应用层：Vue 3 + Vite，负责 NFC 路由、表单、仪表盘、SOP 展示、AI 问答界面
   ↓
后端 API 服务层：NestJS，负责鉴权、学生档案、导师匹配、SOP、任务、通知、文件处理
   ↓
业务数据库：PostgreSQL + Prisma，保存 NFC 卡片、学生、导师、SOP、任务、打卡、通知、问答日志
   ↓
AI 与知识库层：Dify 应用、Dify 知识库、文档摘要、检索问答、规划生成
```

### 4.2 前端

前端包含三个主要端：

- 学生移动端 H5：承接 NFC 碰卡入口，完成首次激活、资料上传、目标选择、导师就位展示、个人仪表盘、AI 问答、打卡和目标修订。
- 导师工作台：用于查看学生档案、阅读资料摘要、确认 SOP、维护任务、处理目标修订和记录沟通情况。
- 管理后台：用于维护 NFC 卡片、学校资料、Dify 知识库配置、导师信息、学生状态、通知模板、奖金规则和运营数据。

前端技术栈采用 Vue 3 + Vite + TypeScript。学生端、导师工作台和管理后台可放在同一前端工程内，通过路由和权限区分不同入口；也可以在后续业务扩大后拆分为多个前端应用。第一阶段建议保持一个前端工程，降低部署和维护成本。

前端不直接访问 Dify，也不直接读写数据库。所有业务请求统一通过后端 API 完成，避免 API Key 暴露和越权访问。

### 4.3 后端

后端是系统的业务中枢，主要职责包括：

- NFC 路由解析：根据卡片 ID、批次 ID 和卡片状态决定学生进入哪个页面。
- 用户与权限管理：区分学生、导师、管理员权限，控制资料、SOP、问答和任务的可见范围。
- 学生档案管理：保存基础信息、目标标签、上传资料、导师关系和陪跑状态。
- 文件处理与异步任务：接收资料上传，触发解析、摘要、入库、失败重试和状态通知。
- SOP 与任务服务：管理 SOP 版本、导师确认、任务拆解、打卡记录和红线预警。
- Dify 调用代理：统一封装 Dify API 调用，注入学生上下文，控制知识库范围，校验返回格式。
- 通知服务：在激活成功、导师匹配、资料解析完成、SOP 确认、任务更新等节点发送短信、邮件或站内通知。

后端技术栈采用 NestJS + TypeScript。后端以模块化方式组织 NFC、学生、导师、SOP、任务、文件、AI、通知和管理后台等业务域，统一提供 REST API。Dify API Key、数据库连接和第三方服务密钥均只保存在后端环境变量中。

### 4.4 数据库

数据库保存结构化业务数据，是系统状态判断和权限控制的基础。核心数据包括：

- NFC 卡片：卡片 ID、批次、绑定状态、绑定学生、激活时间。
- 学生档案：姓名、联系方式、学校、专业、年级、目标、导师关系、陪跑状态。
- 导师档案：姓名、标签、擅长方向、服务学生、沟通记录。
- SOP 版本：版本号、状态、生成来源、导师确认人、历史版本。
- 任务与打卡：学期任务、优先级、截止时间、完成状态、连续打卡记录。
- 资料与知识库映射：上传文件、解析状态、Dify 文档 ID、摘要、权限范围。
- 问答与通知日志：AI 问答记录、引用来源、导师转人工标记、通知发送结果。

数据库采用 PostgreSQL，数据访问层采用 Prisma。数据库只保存可结构化管理的数据。大文件原件建议放入对象存储或文件服务，数据库保存文件地址、状态和 Dify 关联 ID。

### 4.5 知识库

知识库用于支撑 AI 问答、资料摘要、规则提取、SOP 草案生成和目标修订分析。知识库按权限和内容来源分层：

- 学校公共知识库：学生手册、培养方案、评奖评优政策、专业介绍等，同校或同专业学生可访问。
- 学生私有知识库：学生上传的个人资料、成绩、目标记录、SOP 历史，仅学生本人、匹配导师和管理员可访问。
- 平台方法论知识库：保研、考研、竞赛、科研、就业、出国等通用方法论，全平台可复用。
- 导师知识库：导师个人经验、模板、案例复盘和辅导策略，按导师授权范围使用。

知识库由后端统一调用，前端只看到经过权限过滤和格式整理后的问答结果、摘要和行动建议。涉及学校政策、升学资格、奖金金额等高风险内容时，AI 输出必须提示来源并保留人工确认入口。

### 4.6 核心数据流

首次激活数据流：

```text
NFC 碰卡
   ↓
前端读取 nfc_id / batch_id
   ↓
后端校验卡片状态并创建学生档案
   ↓
资料上传至文件服务，元数据写入数据库
   ↓
后端触发 Dify 文档解析和摘要
   ↓
生成 V0.1 SOP 草案并等待导师确认
```

日常问答数据流：

```text
学生在仪表盘提问
   ↓
前端提交问题和会话 ID
   ↓
后端读取学生档案、SOP、任务和权限范围
   ↓
后端调用 Dify 检索可访问知识库
   ↓
后端校验回答、补充引用来源和风险提示
   ↓
前端展示答案，并记录问答日志
```

导师确认 SOP 数据流：

```text
导师打开学生档案
   ↓
后端返回学生资料摘要、V0.1 SOP 草案和关键风险
   ↓
导师调整路线图和任务
   ↓
后端保存 SOP V1.0 版本
   ↓
系统通知学生再次碰卡查看导师方案
```

## 5. 核心业务流程

### 5.1 第一次碰卡：初见，开启旅程

触发条件：

- 学生收到录取通知书。
- NFC 卡片尚未绑定学生。
- NFC 链接中的卡片 ID 未完成激活。

页面流程：

1. 进入欢迎页，展示全屏动效和文案“欢迎开启你的大学旅程”。
2. 跳转至信息提交页。
3. 学生填写姓名、手机号、邮箱、录取院校、专业等基础信息。
4. 学生上传学生手册、评优评奖政策、培养方案等资料。
5. 学生选择未来目标，可多选：
   - 保研/推免
   - 考研深造
   - 大满贯毕业
   - 高质量就业
   - 出国留学
   - 考公考编
   - 自定义目标
6. 系统保存资料，卡片状态变为“待匹配导师”。
7. 后端调用 Dify，对上传资料进行摘要、规则提取和初版 SOP 生成。

输出结果：

- 学生档案创建完成。
- 资料进入知识库处理队列。
- 生成 V0.1 SOP 草案。
- 页面进入等待导师匹配状态。

### 5.2 第二次碰卡：连接，导师就位

触发条件：

- 学生已完成激活。
- 系统已完成导师匹配。
- 导师已初步查看学生资料，或已确认可联系学生。

页面流程：

1. 学生再次触碰 NFC 卡片。
2. 页面展示“信件拆开”或“解锁”动效。
3. 展示导师名片：
   - 头像
   - 姓名
   - 资历标签
   - 擅长方向
   - 过往战绩
   - 一键联系按钮
4. 展示 V1.0 SOP 概览，或展示“导师正在完善你的专属路线图”。
5. 学生可预约首次沟通。

系统触达：

- 匹配完成后，系统通过短信或邮件通知学生：
  “【OPC陪跑】同学你好，你的专属指导老师已就位！请再次触碰你的 NFC 卡片查看。”

输出结果：

- 学生与导师建立关系。
- SOP 从 V0.1 草案进入导师确认流程。
- 学生进入正式陪跑阶段。

### 5.3 第三次及以后碰卡：启程，日常陪跑

触发条件：

- 学生已绑定。
- 导师已确认或正在维护 SOP。
- 学生进入日常学习阶段。

页面流程：

1. 学生碰卡后直接进入个人仪表盘。
2. 查看当前阶段目标、本周任务、红线预警、打卡进度和奖金收益。
3. 可向 AI 助手提问，例如：
   - “我这个专业保研要重点关注哪些指标？”
   - “大一上应该优先做什么？”
   - “这份培养方案里有哪些容易踩坑的规则？”
4. 前端将问题和学生上下文发送给后端，后端调用 Dify 知识库返回答案。
5. 学生完成任务后可打卡，系统记录连续打卡天数。

输出结果：

- 学生形成持续行动记录。
- 导师可基于进度调整任务。
- 系统积累成长轨迹和 SOP 版本历史。

## 6. 页面设计

### 6.1 NFC 路由入口页

访问地址示例：

```text
https://e.yayasmart.com?nfc_id=xxx&batch_id=xxx
```

路由规则：

| 卡片状态 | 页面去向 |
| --- | --- |
| 未识别或参数缺失 | 异常提示页 |
| 未绑定学生 | 首次激活页 |
| 已绑定但未匹配导师 | 等待匹配页 |
| 已匹配导师但 SOP 未确认 | 导师就位页 |
| SOP 已确认 | 学生仪表盘 |

### 6.2 首次激活页

核心模块：

- 欢迎动效
- 基础信息表单
- 目标选择器
- 资料上传区
- 隐私授权确认
- 提交按钮

设计重点：

- 移动端优先。
- 表单步骤化，降低一次性填写压力。
- 上传资料支持拍照、相册选择和文件上传。
- 提交后即刻给出反馈，Dify 分析可异步进行。

### 6.3 等待匹配页

核心模块：

- 当前状态：“你的资料已收到”
- AI 解析进度
- 导师匹配状态
- 预计通知方式
- 可补充资料入口

容错文案：

- “导师正在赶来的路上”
- “你的专属攻略正在生成中”
- “资料较多时可能需要多一点时间”

### 6.4 导师就位页

核心模块：

- 解锁动效
- 导师名片
- 一键联系
- 预约沟通
- SOP 概览
- 下一步提示

导师名片字段：

- 姓名
- 头像
- 标签，如“985硕士”“前大厂高工”“竞赛金牌教练”
- 擅长目标，如保研、竞赛、科研、就业
- 辅导次数
- 好评率
- 学员成果

### 6.5 学生仪表盘

核心模块：

- 今日状态
- 当前学期目标
- 本周任务
- SOP 通关地图
- 红线预警
- AI 问答入口
- 连续打卡天数
- 奖金收益
- 荣誉排行榜
- 目标修订入口

设计重点：

- 首页直接展示最该做的事。
- 不做营销式首页，第一屏即进入学生自己的任务与进度。
- 关键风险信息醒目展示。
- AI 问答入口固定可见，但不喧宾夺主。

### 6.6 SOP 规划详情页

核心模块：

- 四年时间轴
- 按学期拆解的任务清单
- 当前行动指南
- 红线预警
- SOP 版本记录
- 导师备注

任务字段：

- 任务名称
- 所属学期
- 优先级
- 截止时间
- 完成标准
- 关联目标
- 关联资料来源
- 导师备注
- 完成状态

### 6.7 AI 知识库问答页

核心模块：

- 对话窗口
- 推荐问题
- 引用来源
- 学生上下文提示
- 转人工导师按钮

问答来源：

- 学生上传的培养方案、学生手册、评奖规则。
- 平台维护的通用升学、就业、竞赛、科研知识。
- 导师沉淀的方法论文档。
- 学生当前 SOP、任务和目标。

回答要求：

- 明确区分“资料中有依据”和“系统推断建议”。
- 对学校政策类问题尽量返回来源文件名称。
- 对高风险事项提示学生找辅导员、导师或学校官方确认。

### 6.8 目标修订页

核心模块：

- 当前目标
- 修订原因
- 新目标选择
- 影响评估
- 提交导师确认
- 历史版本记录

流程规则：

- 学生可发起目标修订。
- 系统调用 Dify 生成初步影响分析。
- 导师二次沟通后确认新版本 SOP。
- 旧 SOP 保留为历史版本，不覆盖删除。

### 6.9 导师工作台

核心模块：

- 待处理学生列表
- 学生档案详情
- 上传资料摘要
- Dify 辅助分析结果
- SOP 编辑器
- 任务维护
- 沟通记录
- 目标修订审核

导师操作：

- 查看学生目标与基础信息。
- 查看 Dify 对学校资料的摘要。
- 生成或调整四年路线图。
- 确认 SOP 版本。
- 更新本周任务。
- 标记风险和备注。

### 6.10 管理后台

核心模块：

- NFC 卡片管理
- 学校资料库管理
- Dify 知识库配置
- 导师管理
- 学生管理
- 奖金规则管理
- 通知模板管理
- 数据看板

## 7. Dify 后端知识库接入设计

### 7.1 接入目标

Dify 负责提供知识库检索、文档问答、摘要生成和规划建议能力。网站后端不直接把 Dify 暴露给前端，而是通过自有服务层统一鉴权、记录上下文、处理敏感信息和控制返回格式。

### 7.2 推荐架构

```text
学生/导师前端
   ↓
网站后端 API
   ↓
业务服务层：鉴权、学生上下文、资料权限、日志
   ↓
Dify API：知识库检索、聊天应用、工作流应用
   ↓
Dify 知识库：学校资料、学生资料、平台资料、导师资料
```

### 7.3 知识库分类

| 知识库 | 内容 | 权限 |
| --- | --- | --- |
| 学校公共知识库 | 学生手册、培养方案、评奖政策、专业介绍 | 同校学生可用 |
| 学生私有知识库 | 学生上传的个人资料、成绩、目标记录 | 学生本人、匹配导师、管理员可用 |
| 平台方法论知识库 | 保研、考研、就业、竞赛、科研方法论 | 全平台可用 |
| 导师知识库 | 导师个人经验、辅导模板、案例复盘 | 导师本人、授权学生可用 |

### 7.4 主要调用场景

#### 场景一：资料上传后自动摘要

输入：

- 学校名称
- 专业名称
- 学生目标
- 上传文件

输出：

- 文件摘要
- 关键规则
- 风险项
- 与学生目标相关的机会点
- V0.1 SOP 建议

#### 场景二：学生日常提问

输入：

- 学生问题
- 学生学校、专业、年级、目标
- 当前 SOP
- 可访问知识库范围

输出：

- 直接回答
- 引用来源
- 行动建议
- 是否建议转导师

#### 场景三：导师生成 SOP

输入：

- 学生档案
- 目标标签
- Dify 资料摘要
- 导师补充要求

输出：

- 四年路线图
- 学期任务
- 当前阶段行动指南
- 红线预警
- 需要导师确认的疑点

#### 场景四：目标修订影响分析

输入：

- 原目标
- 新目标
- 当前进度
- 学生已完成成果

输出：

- 调整可行性
- 需要补齐的能力
- 已有任务影响
- 新 SOP 草案

### 7.5 后端 API 设计

前端调用网站后端，网站后端再调用 Dify。

```text

POST /api/ai/chat
POST /api/ai/generate-sop
POST /api/ai/revise-goal
```

示例：学生 AI 问答接口

```json

  "student_id": "stu_001",
  "conversation_id": "conv_001",
  "question": "我大一上想保研，应该优先做什么？",
  "context": {
    "school": "某某大学",
    "major": "计算机科学与技术",
    "grade": "大一",
    "goals": ["保研/推免", "竞赛"]
  }
}
```

后端返回：

```json
{
  "answer": "建议优先稳定 GPA、完成英语四级准备，并尽早了解学院保研细则。",
  "sources": [
    {
      "name": "本科生培养方案.pdf",
      "section": "毕业与学位要求"
    }
  ],
  "actions": [
    "本周整理专业必修课清单",
    "确认本学期绩点目标",
    "预约导师沟通保研路径"
  ],
  "need_mentor": false

```

### 7.6 Dify 调用注意事项

- Dify API Key 只保存在服务端环境变量中，不写入前端代码。
- 每次请求携带学生 ID、会话 ID 和权限范围，但避免传入不必要的敏感信息。
- 对 Dify 返回内容做格式校验，避免直接渲染不可信 HTML。
- 对政策类、金额类、升学资格类问题增加免责声明和人工确认提示。
- 存储问答日志，便于导师追踪学生关注点。
- 对文件处理采用异步队列，前端显示处理中状态。

## 9. 权限与隐私

- 学生只能访问自己的档案、SOP、问答记录和私有资料。
- 导师只能访问已分配给自己的学生。
- 管理员可访问全局数据，但关键操作需要日志记录。
- 学生上传资料前需要勾选隐私授权。
- 敏感字段如手机号、邮箱、成绩单等需要脱敏展示。
- Dify 知识库需按学校、学生、导师进行隔离，避免跨学生泄露私有资料。

## 12. 体验原则

- NFC 入口要快，打开后 3 秒内给出明确反馈。
- 首次表单不要过长，资料可后续补充。
- Dify 分析耗时较长时使用异步状态，不阻塞用户。
- 重大节点使用全屏动效增强仪式感。
- 错误信息使用陪伴式文案，避免冷冰冰的失败提示。
- 日常仪表盘第一屏聚焦“今天该做什么”。
- 所有 AI 建议都需要为导师确认留出空间。

## 15. V2.4 修订基线

本节为依据《产品执行手册 V2.4》和 `yayasmart-demo` 补充的正式修订。若前文与本节冲突，以本节为准。

### 15.1 产品范围修订

系统不再只覆盖学生、导师、管理员三类入口，正式范围包括：

- 学生端：激活建档、SOP、任务执行、成果提交、导师反馈、复盘、问答、荣誉、奖金和目标修订。
- 家长端：通过亲情卡进入只读观察室，查看经学生授权的进度、荣誉和导师评价摘要，并发送鼓励。
- 导师端：确认 SOP、发布任务、验收成果、退回补充、风险干预、阶段复盘和目标修订审核。
- 运营与学校角色端：项目老师、辅导员、学校老师、专属客服按负责范围查看和处理学生事项。
- 管理端：卡片全生命周期、多卡绑定、用户与角色、导师分配、知识库、通知、奖金审核和审计。

### 15.2 学生全周期闭环

学生系统必须形成以下可追踪闭环，不能把“勾选任务完成”作为终点：

```text
NFC 激活建档
  → 资料解析与 SOP V0.1
  → 导师匹配、沟通与合作模式确认
  → 导师确认 SOP V1.0
  → 系统按周发布任务
  → 学生执行并提交文字/图片/文件/链接等成果证据
  → 导师验收通过或退回补充
  → 更新任务进度、风险、连续行动和成长事件
  → 周复盘/月复盘/学期复盘
  → 解锁荣誉，审核奖金或生成待结算流水
  → 目标变化时发起修订并生成新 SOP
  → 新周期继续执行
```

闭环中的每一步都必须有状态、责任人、时间戳、通知和可恢复入口。学生仪表盘第一屏按优先级展示“待补交成果、被退回任务、即将截止任务、今日任务”，而不是只展示统计数字。

### 15.3 学生端页面信息架构

| 一级入口 | 页面/抽屉 | 核心动作 |
| --- | --- | --- |
| 首页 | 学生仪表盘 | 查看今日行动、预警、导师反馈和待处理事项 |
| 规划 | SOP 通关地图、版本历史 | 查看路线图、来源和版本变更 |
| 任务 | 任务列表、任务详情、成果提交 | 开始任务、上传证据、补充材料、查看验收 |
| 复盘 | 周/月/学期复盘 | 自评、总结阻塞、确认下一周期重点 |
| AI | 知识库问答 | 获取有来源的建议并可转导师 |
| 成长 | 成长档案、荣誉、奖金流水 | 查看已验收成果、徽章和财务状态 |
| 消息 | 导师/家长/系统消息 | 接收退回、通过、提醒和鼓励 |
| 我的 | 资料、目标、卡片、隐私授权 | 管理账号、多卡、家长授权和目标修订 |

### 15.4 NFC 多卡与角色路由

V2.4 的物理卡标识使用：

- `idd`：NFC 物理卡唯一标识。
- `idh`：8 位十六进制业务编号，用于角色范围和卡片身份映射。

为兼容已有链接，入口可继续接受 `id1`、`id2`，但服务端统一转换为 `idd`、`idh`，业务表和 API 返回不再使用“批次 ID 代表用户身份”的含混定义。

同一学生可绑定多张卡：

- 学生主卡：进入学生当前状态页面。
- 学生副卡：与主卡相同权限，可单独挂失。
- 父母亲情卡：进入家长观察室，不可进入学生私有操作页。

卡片状态和学生陪跑状态必须分离。卡片的 `unbound/active/lost/replaced/revoked` 描述物理生命周期；学生的 `activating/pending_match/mentor_matched/sop_reviewing/active/paused/graduated` 描述业务生命周期。

### 15.5 家长观察室

家长端沿用 `yayasmart-demo/parent.html` 的“只读观察室 + 发送鼓励”结构，正式实现时增加：

- 首次使用需完成手机号验证并接受亲情卡授权说明。
- 学生可选择是否展示具体任务名称、导师评价摘要和荣誉详情。
- 默认不展示成绩单原件、私人问答、联系方式、奖金分润细节和上传文件。
- 鼓励消息进入消息中心，支持署名或匿名；系统记录发送状态并提供频率限制。
- 学生撤回授权、亲情卡挂失或关系解除后立即失去访问权限。

## 16. 统一验收场景

至少通过以下端到端场景：

1. 新学生碰主卡，完成建档和资料上传，刷新后草稿与上传结果仍存在。
2. 导师分配前再次碰卡进入等待页；分配后进入导师就位页。
3. 导师确认 SOP V1.0 后，学生碰卡进入仪表盘并看到本周任务。
4. 学生提交任务成果，导师退回，学生补充后再次提交并通过验收。
5. 通过验收后任务进度、成长档案和相关荣誉条件同步更新。
6. 学生完成周复盘，导师给出反馈并生成下一周任务。
7. 学生发起目标修订，旧 SOP 保持生效，导师确认后新版本切换生效。
8. 父母碰亲情卡只能看到授权摘要，可发送鼓励但不可修改任务。
9. 学生撤回家长授权后，亲情卡再次访问被拒绝。
10. 主卡挂失后不可访问；副卡仍可用；换卡后新主卡接续原学生数据。
