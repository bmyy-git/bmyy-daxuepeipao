# 笨猫丫丫 OPC 学习陪跑网站设计文档

## 1. 项目概述

笨猫丫丫 OPC 学习陪跑网站是一个以 NFC 卡片为入口、以 Dify 后端知识库和导师服务为支撑的大学四年学业陪跑平台。学生通过“碰一碰”打开专属网页，从录取后首次激活开始，逐步完成信息录入、资料上传、AI 初步规划、导师匹配、SOP 确认、日常打卡和成长复盘。

产品核心定位：

> 一张 NFC 卡片，开启四年学业陪跑。

网站不只是一个表单或资料查询工具，而是连接学生从“收到录取通知书”到“大学四年成长”的数字入口。它通过“线下 NFC 触发 + 线上动态规划 + Dify 知识库问答 + 导师人工校准”，提供有仪式感、可迭代、可跟踪的长期陪伴体验。

## 2. 建设目标

- 为每张 NFC 卡片建立唯一数字身份，支持不同状态下的自动路由。
- 让新生完成学校、专业、目标、资料等基础信息提交。
- 接入 Dify 后端知识库，对学校政策、培养方案、学生手册、评奖规则等资料进行问答和规划辅助。
- 生成学生个人 V0.1 SOP 草案，并支持导师确认为 V1.0。
- 提供学生专属仪表盘，展示四年路线图、本周任务、红线预警、奖金收益、荣誉榜和打卡记录。
- 支持学生在大学不同阶段修订目标，保留 SOP 历史版本。
- 为导师提供基础工作台，支持查看学生资料、调用知识库辅助分析、确认规划和更新任务。

## 3. 用户角色

### 3.1 学生

主要使用移动端网页访问。通过 NFC 卡片进入网站，完成激活、提交资料、查看规划、日常打卡、提问和目标修订。

### 3.2 导师

主要使用桌面端或平板端后台。查看系统分配的学生，阅读学生上传资料和 AI 摘要，结合自身经验确认 SOP、维护任务、处理目标修订。

### 3.3 管理员

维护 NFC 卡片、学校知识库、导师资料、学生状态、奖金规则、内容审核和系统配置。

### 3.4 Dify 知识库服务

作为 AI 能力和知识库检索后端，负责基于已上传或已维护的学校资料、平台规则、导师方法论等内容，为前端和后台提供问答、摘要、规划建议和任务拆解能力。

## 4. 核心业务流程

### 4.1 第一次碰卡：初见，开启旅程

触发条件：

- 学生收到录取通知书。
- NFC 卡片尚未绑定学生。
- NFC 链接中的卡片 ID 未完成激活。

页面流程：

1. 进入欢迎页，展示全屏动效和文案“欢迎开启你的大学旅程”。
2. 跳转至信息提交页。
3. 学生填写姓名、手机号、邮箱、录取院校、专业等基础信息。
4. 学生上传学生手册、评优评奖政策、培养方案等资料。
5. 学生选择未来目标，可多选：
   - 保研/推免
   - 考研深造
   - 大满贯毕业
   - 高质量就业
   - 出国留学
   - 考公考编
   - 自定义目标
6. 系统保存资料，卡片状态变为“待匹配导师”。
7. 后端调用 Dify，对上传资料进行摘要、规则提取和初版 SOP 生成。

输出结果：

- 学生档案创建完成。
- 资料进入知识库处理队列。
- 生成 V0.1 SOP 草案。
- 页面进入等待导师匹配状态。

### 4.2 第二次碰卡：连接，导师就位

触发条件：

- 学生已完成激活。
- 系统已完成导师匹配。
- 导师已初步查看学生资料，或已确认可联系学生。

页面流程：

1. 学生再次触碰 NFC 卡片。
2. 页面展示“信件拆开”或“解锁”动效。
3. 展示导师名片：
   - 头像
   - 姓名
   - 资历标签
   - 擅长方向
   - 过往战绩
   - 一键联系按钮
4. 展示 V1.0 SOP 概览，或展示“导师正在完善你的专属路线图”。
5. 学生可预约首次沟通。

系统触达：

- 匹配完成后，系统通过短信或邮件通知学生：
  “【OPC陪跑】同学你好，你的专属指导老师已就位！请再次触碰你的 NFC 卡片查看。”

输出结果：

- 学生与导师建立关系。
- SOP 从 V0.1 草案进入导师确认流程。
- 学生进入正式陪跑阶段。

### 4.3 第三次及以后碰卡：启程，日常陪跑

触发条件：

- 学生已绑定。
- 导师已确认或正在维护 SOP。
- 学生进入日常学习阶段。

页面流程：

1. 学生碰卡后直接进入个人仪表盘。
2. 查看当前阶段目标、本周任务、红线预警、打卡进度和奖金收益。
3. 可向 AI 助手提问，例如：
   - “我这个专业保研要重点关注哪些指标？”
   - “大一上应该优先做什么？”
   - “这份培养方案里有哪些容易踩坑的规则？”
4. 前端将问题和学生上下文发送给后端，后端调用 Dify 知识库返回答案。
5. 学生完成任务后可打卡，系统记录连续打卡天数。

输出结果：

- 学生形成持续行动记录。
- 导师可基于进度调整任务。
- 系统积累成长轨迹和 SOP 版本历史。

## 5. 页面设计

### 5.1 NFC 路由入口页

访问地址示例：

```text
https://e.yayasmart.com?nfc_id=xxx&batch_id=xxx
```

路由规则：

| 卡片状态 | 页面去向 |
| --- | --- |
| 未识别或参数缺失 | 异常提示页 |
| 未绑定学生 | 首次激活页 |
| 已绑定但未匹配导师 | 等待匹配页 |
| 已匹配导师但 SOP 未确认 | 导师就位页 |
| SOP 已确认 | 学生仪表盘 |

### 5.2 首次激活页

核心模块：

- 欢迎动效
- 基础信息表单
- 目标选择器
- 资料上传区
- 隐私授权确认
- 提交按钮

设计重点：

- 移动端优先。
- 表单步骤化，降低一次性填写压力。
- 上传资料支持拍照、相册选择和文件上传。
- 提交后即刻给出反馈，Dify 分析可异步进行。

### 5.3 等待匹配页

核心模块：

- 当前状态：“你的资料已收到”
- AI 解析进度
- 导师匹配状态
- 预计通知方式
- 可补充资料入口

容错文案：

- “导师正在赶来的路上”
- “你的专属攻略正在生成中”
- “资料较多时可能需要多一点时间”

### 5.4 导师就位页

核心模块：

- 解锁动效
- 导师名片
- 一键联系
- 预约沟通
- SOP 概览
- 下一步提示

导师名片字段：

- 姓名
- 头像
- 标签，如“985硕士”“前大厂高工”“竞赛金牌教练”
- 擅长目标，如保研、竞赛、科研、就业
- 辅导次数
- 好评率
- 学员成果

### 5.5 学生仪表盘

核心模块：

- 今日状态
- 当前学期目标
- 本周任务
- SOP 通关地图
- 红线预警
- AI 问答入口
- 连续打卡天数
- 奖金收益
- 荣誉排行榜
- 目标修订入口

设计重点：

- 首页直接展示最该做的事。
- 不做营销式首页，第一屏即进入学生自己的任务与进度。
- 关键风险信息醒目展示。
- AI 问答入口固定可见，但不喧宾夺主。

### 5.6 SOP 规划详情页

核心模块：

- 四年时间轴
- 按学期拆解的任务清单
- 当前行动指南
- 红线预警
- SOP 版本记录
- 导师备注

任务字段：

- 任务名称
- 所属学期
- 优先级
- 截止时间
- 完成标准
- 关联目标
- 关联资料来源
- 导师备注
- 完成状态

### 5.7 AI 知识库问答页

核心模块：

- 对话窗口
- 推荐问题
- 引用来源
- 学生上下文提示
- 转人工导师按钮

问答来源：

- 学生上传的培养方案、学生手册、评奖规则。
- 平台维护的通用升学、就业、竞赛、科研知识。
- 导师沉淀的方法论文档。
- 学生当前 SOP、任务和目标。

回答要求：

- 明确区分“资料中有依据”和“系统推断建议”。
- 对学校政策类问题尽量返回来源文件名称。
- 对高风险事项提示学生找辅导员、导师或学校官方确认。

### 5.8 目标修订页

核心模块：

- 当前目标
- 修订原因
- 新目标选择
- 影响评估
- 提交导师确认
- 历史版本记录

流程规则：

- 学生可发起目标修订。
- 系统调用 Dify 生成初步影响分析。
- 导师二次沟通后确认新版本 SOP。
- 旧 SOP 保留为历史版本，不覆盖删除。

### 5.9 导师工作台

核心模块：

- 待处理学生列表
- 学生档案详情
- 上传资料摘要
- Dify 辅助分析结果
- SOP 编辑器
- 任务维护
- 沟通记录
- 目标修订审核

导师操作：

- 查看学生目标与基础信息。
- 查看 Dify 对学校资料的摘要。
- 生成或调整四年路线图。
- 确认 SOP 版本。
- 更新本周任务。
- 标记风险和备注。

### 5.10 管理后台

核心模块：

- NFC 卡片管理
- 学校资料库管理
- Dify 知识库配置
- 导师管理
- 学生管理
- 奖金规则管理
- 通知模板管理
- 数据看板

## 6. Dify 后端知识库接入设计

### 6.1 接入目标

Dify 负责提供知识库检索、文档问答、摘要生成和规划建议能力。网站后端不直接把 Dify 暴露给前端，而是通过自有服务层统一鉴权、记录上下文、处理敏感信息和控制返回格式。

### 6.2 推荐架构

```text
学生/导师前端
   ↓
网站后端 API
   ↓
业务服务层：鉴权、学生上下文、资料权限、日志
   ↓
Dify API：知识库检索、聊天应用、工作流应用
   ↓
Dify 知识库：学校资料、学生资料、平台资料、导师资料
```

### 6.3 知识库分类

| 知识库 | 内容 | 权限 |
| --- | --- | --- |
| 学校公共知识库 | 学生手册、培养方案、评奖政策、专业介绍 | 同校学生可用 |
| 学生私有知识库 | 学生上传的个人资料、成绩、目标记录 | 学生本人、匹配导师、管理员可用 |
| 平台方法论知识库 | 保研、考研、就业、竞赛、科研方法论 | 全平台可用 |
| 导师知识库 | 导师个人经验、辅导模板、案例复盘 | 导师本人、授权学生可用 |

### 6.4 主要调用场景

#### 场景一：资料上传后自动摘要

输入：

- 学校名称
- 专业名称
- 学生目标
- 上传文件

输出：

- 文件摘要
- 关键规则
- 风险项
- 与学生目标相关的机会点
- V0.1 SOP 建议

#### 场景二：学生日常提问

输入：

- 学生问题
- 学生学校、专业、年级、目标
- 当前 SOP
- 可访问知识库范围

输出：

- 直接回答
- 引用来源
- 行动建议
- 是否建议转导师

#### 场景三：导师生成 SOP

输入：

- 学生档案
- 目标标签
- Dify 资料摘要
- 导师补充要求

输出：

- 四年路线图
- 学期任务
- 当前阶段行动指南
- 红线预警
- 需要导师确认的疑点

#### 场景四：目标修订影响分析

输入：

- 原目标
- 新目标
- 当前进度
- 学生已完成成果

输出：

- 调整可行性
- 需要补齐的能力
- 已有任务影响
- 新 SOP 草案

### 6.5 后端 API 设计

前端调用网站后端，网站后端再调用 Dify。

```text
POST /api/ai/analyze-documents
POST /api/ai/chat
POST /api/ai/generate-sop
POST /api/ai/revise-goal
```

示例：学生 AI 问答接口

```json
{
  "student_id": "stu_001",
  "conversation_id": "conv_001",
  "question": "我大一上想保研，应该优先做什么？",
  "context": {
    "school": "某某大学",
    "major": "计算机科学与技术",
    "grade": "大一",
    "goals": ["保研/推免", "竞赛"]
  }
}
```

后端返回：

```json
{
  "answer": "建议优先稳定 GPA、完成英语四级准备，并尽早了解学院保研细则。",
  "sources": [
    {
      "name": "本科生培养方案.pdf",
      "section": "毕业与学位要求"
    }
  ],
  "actions": [
    "本周整理专业必修课清单",
    "确认本学期绩点目标",
    "预约导师沟通保研路径"
  ],
  "need_mentor": false
}
```

### 6.6 Dify 调用注意事项

- Dify API Key 只保存在服务端环境变量中，不写入前端代码。
- 每次请求携带学生 ID、会话 ID 和权限范围，但避免传入不必要的敏感信息。
- 对 Dify 返回内容做格式校验，避免直接渲染不可信 HTML。
- 对政策类、金额类、升学资格类问题增加免责声明和人工确认提示。
- 存储问答日志，便于导师追踪学生关注点。
- 对文件处理采用异步队列，前端显示处理中状态。

## 8. 权限与隐私

- 学生只能访问自己的档案、SOP、问答记录和私有资料。
- 导师只能访问已分配给自己的学生。
- 管理员可访问全局数据，但关键操作需要日志记录。
- 学生上传资料前需要勾选隐私授权。
- 敏感字段如手机号、邮箱、成绩单等需要脱敏展示。
- Dify 知识库需按学校、学生、导师进行隔离，避免跨学生泄露私有资料。

## 9. 通知机制

通知渠道：

关键通知节点：

- 学生激活成功
- 资料解析完成
- 导师匹配完成
- SOP V1.0 已确认
- 本周任务更新
- 红线风险提醒
- 目标修订结果

## 10. 奖金与荣誉模块

### 10.1 奖金模块

展示内容：

- 累计获得总额
- 学生实得金额
- 导师分润金额
- 平台规则说明
- 交易明细

注意事项：

- 奖金和分润规则涉及财务合规，页面需以平台确认规则为准。
- 对“对赌”“分润”等表达需根据实际商业合同谨慎调整。
- 金额类数据必须有审核流程，不建议完全自动入账。

### 10.2 荣誉体系

展示内容：

- 排行榜
- 连续打卡榜
- 大满贯毕业生案例
- 学期成就徽章

设计原则：

- 激励学生持续行动。
- 避免过度制造焦虑。
- 排行榜可支持匿名或昵称展示。

## 11. 体验原则

- NFC 入口要快，打开后 3 秒内给出明确反馈。
- 首次表单不要过长，资料可后续补充。
- Dify 分析耗时较长时使用异步状态，不阻塞用户。
- 重大节点使用全屏动效增强仪式感。
- 错误信息使用陪伴式文案，避免冷冰冰的失败提示。
- 日常仪表盘第一屏聚焦“今天该做什么”。
- 所有 AI 建议都需要为导师确认留出空间。
