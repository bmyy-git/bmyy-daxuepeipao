# 详细设计-02-学生激活与资料上传

## 1. 模块目标

学生激活与资料上传模块负责完成第一次碰卡后的信息收集，为后续 Dify 分析、导师匹配和 SOP 草案生成提供基础数据。

该模块对应“第一幕：初见·开启旅程”。

## 2. 业务流程

```text
学生第一次碰卡
  ↓
进入欢迎动效
  ↓
填写基础信息
  ↓
选择未来目标
  ↓
上传学校和专业资料
  ↓
勾选隐私授权
  ↓
提交激活
  ↓
绑定 NFC 卡片
  ↓
进入等待匹配页
```

## 3. 页面步骤

## 5. 文件上传安全与可靠性

文件上传是学生私有资料进入系统和 Dify 知识库的入口，必须按“先校验、再存储、后解析”的原则实现。

### 5.1 上传基本规则

| 项目 | 规则 |
| --- | --- |
| 单文件大小 | MVP 建议不超过 20MB |
| 单次上传数量 | MVP 建议不超过 10 个文件 |
| 支持格式 | pdf / doc / docx / jpg / jpeg / png |
| 存储权限 | 私有存储，不允许公开永久 URL |
| 解析方式 | 异步解析，不能阻塞激活流程 |
| 文件归属 | 激活前归属 `activation_session_id`，激活后绑定 `student_id` |

### 5.2 激活前临时归属

学生在正式创建档案前也可能先上传文件，因此必须引入 `activation_session_id`。

```text
进入首次激活页
  ↓
后端创建 activation_session_id
  ↓
上传文件时携带 activation_session_id、card_id、batch_id
  ↓
后端保存文件元数据，owner_type = activation_session
  ↓
学生提交激活
  ↓
后端事务内创建 student，并将文件 owner_type 改为 student
```

`activation_session_id` 必须由后端生成，避免前端伪造过多无效会话。

### 5.3 后端校验要求

上传接口必须做以下校验：

- 校验 `card_id`、`batch_id` 对应卡片存在且未停用。
- 校验 `activation_session_id` 是否有效。
- 校验文件大小。
- 校验文件扩展名。
- 校验文件 MIME 类型。
- 校验文件头魔数，避免伪装扩展名。
- 计算文件 hash，用于去重和审计。
- 对文件名做清洗，不信任用户上传的原始文件名。
- 禁止把文件保存到 Web 可直接访问目录。

如条件允许，生产环境建议增加病毒扫描或恶意文件扫描。MVP 可先保留 `security_scan_status` 字段，默认走异步扫描或人工兜底。

### 5.4 上传状态

| 状态 | 含义 |
| --- | --- |
| uploading | 上传中 |
| uploaded | 已上传，等待校验或解析 |
| validating | 校验中 |
| rejected | 文件不符合规则 |
| scanning | 安全扫描中 |
| parsing | Dify 解析中 |
| available | 可用 |
| failed | 处理失败，可重试 |

前端只展示面向学生的友好状态，例如“已收到”“正在识别”“需要重新上传”，不直接展示底层错误堆栈。

### 5.5 可靠性要求

- 上传接口必须返回稳定的 `file_id`。
- 前端重复提交同一文件时，后端可通过 hash 去重。
- 文件上传成功但激活提交失败时，文件保持临时归属，不立即删除。
- 临时文件超过保留期后由定时任务清理，MVP 建议保留 7 天。
- Dify 解析失败不影响激活和等待匹配。
- Dify 解析失败的文件必须支持后台重试。
- 文件原件、元数据和解析状态必须分开管理，避免解析失败导致原件丢失。

## 6. 后端接口设计

### 6.1 创建激活会话

```text
POST /api/activation-sessions
```

请求：

```json
{
  "card_id": "card_001",
  "batch_id": "batch_2026"
}
```

返回：

```json
{
  "activation_session_id": "act_001",
  "expires_at": "2026-06-18T12:00:00Z"
}
```

### 6.2 激活学生

```text
POST /api/students/activate
```

请求：

```json
{
  "card_id": "card_001",
  "batch_id": "batch_2026",
  "name": "张三",
  "phone": "13800000000",
  "email": "student@example.com",
  "school": "某某大学",
  "major": "计算机科学与技术",
  "grade": "大一",
  "goals": ["保研/推免", "竞赛"],
  "custom_goal": "",
  "privacy_agreed": true,
  "activation_session_id": "act_001",
  "uploaded_file_ids": ["doc_001", "doc_002"]
}
```

返回：

```json
{
  "student_id": "stu_001",
  "card_status": "pending_match",
  "redirect_to": "waiting"
}
```

### 6.3 上传文件

```text
POST /api/files/upload
```

请求类型：

```text
multipart/form-data
```

必传字段：

| 字段 | 说明 |
| --- | --- |
| activation_session_id | 激活前上传归属 |
| card_id | 卡片 ID |
| batch_id | 批次 ID |
| file | 文件 |

返回：

```json
{
  "file_id": "doc_001",
  "file_name": "培养方案.pdf",
  "status": "uploaded",
  "sha256": "..."
}
```

### 6.4 查询上传文件

```text
GET /api/files?student_id=stu_001
```

## 7. 后端处理逻辑

激活接口必须使用事务：

1. 校验卡片存在。
2. 校验卡片状态为 `unbound`。
3. 创建学生档案。
4. 绑定卡片与学生。
5. 更新卡片状态为 `pending_match`。
6. 关联已上传文件。
7. 创建隐私授权记录。
8. 触发 Dify 文档解析异步任务。
9. 返回等待匹配页。

如果第 3-7 步任意一步失败，事务回滚。

上传接口处理逻辑：

1. 校验激活会话和卡片状态。
2. 校验文件大小、扩展名、MIME 和文件头。
3. 清洗文件名。
4. 计算 hash。
5. 保存文件到私有存储。
6. 写入 documents 元数据。
7. 返回 `file_id`。
8. 异步进入安全扫描和 Dify 解析队列。

## 10. 异常处理

| 场景 | 处理 |
| --- | --- |
| 卡片不存在 | 提示卡片未准备好 |
| 卡片已绑定 | 根据状态跳转到对应页面 |
| 手机号格式错误 | 前端提示重新填写 |
| 文件格式不支持 | 前端提示支持格式 |
| 文件过大 | 提示压缩或拆分上传 |
| 文件头与扩展名不一致 | 拒绝上传并提示文件格式异常 |
| 安全扫描失败 | 标记 rejected，不进入 Dify |
| 临时会话过期 | 提示刷新页面后重新上传 |
| 私有存储写入失败 | 提示稍后重试，不生成可用文档 |
| Dify 解析失败 | 不阻塞激活，进入等待页并标记可重试 |

## 11. 验收标准

- 未绑定卡片可以完成学生激活。
- 激活后卡片状态变为 `pending_match`。
- 学生至少选择一个目标。
- 学生必须勾选隐私授权才能提交。
- 文件上传成功后生成文档记录。
- 文件上传必须绑定 `activation_session_id` 或 `student_id`。
- 不支持的文件格式会被拒绝。
- 超过大小限制的文件会被拒绝。
- 文件不会暴露公开永久 URL。
- 文件解析失败后原文件仍保留，并可后台重试。
- Dify 解析失败不影响学生进入等待匹配页。
- 重复提交不会创建多个学生档案。
